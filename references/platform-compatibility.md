## 模块化导出映射
- 以下名称用于 `design-system-modular.yaml` 的导出索引，方便 AI 或其他引用方直接定位对应的平台规则。
- `dtkFirst`：对应“DTK 优先”。表示桌面组件默认先检查 `org.deepin.dtk` 是否已提供原生能力。
- `customTitlebarDefault`：对应“窗口策略”中的统一头部优先方案。表示主应用窗口优先追求标题栏与工具栏的一体化视觉，但前提是不能破坏窗管拥有的装饰和标准行为。
- `systemDecorationsFallback`：对应“窗口策略”中的默认兼容路径。表示当平台、Qt/DTK 组合无法同时满足统一头部视觉和窗管装饰行为时，退回系统窗口装饰。
- `popupWindowQt68`：对应“弹窗策略”中的窗口管理型弹窗条件。表示仅在 Qt 6.8 及以上、且组件确实需要由窗口管理器单独管理时，再考虑 `Popup.Window`。
- `blurFallback`：对应“模糊与透明”中的 blur 回退规则。表示所有 blur、透明和毛玻璃实现都必须提供纯色背景回退；当 blur 是持久主导航左侧栏的硬要求时，这个回退只是兼容路径，不是可替代的最终设计。

## DTK 优先
- 优先使用 `org.deepin.dtk` 提供的窗口、按钮、菜单、对话框和表单控件。
- 应用主菜单触发按钮与关于对话框属于 DTK 优先的强约束场景；当本地存在 DTK 对应能力时，不要退回自定义按钮或自定义 About 弹窗。
- 只有当 DTK 缺少目标能力，或需要强定制化桌面窗口结构时，才退回自定义 QML 组件。
- 自定义组件必须保留系统主题同步、键盘导航、输入法兼容和无障碍信息。

## 窗口策略
- 标准主窗口默认优先保留由窗管管理的顶层窗口装饰和标准行为，包括边框、阴影、圆角、拖动、缩放、最大化、贴边和相关桌面交互。
- 如果目标 Qt/DTK/桌面环境已经验证支持“统一头部视觉”且不破坏上述窗管能力，可以在系统装饰语义之上实现标题栏与工具栏的一体化呈现。
- 如果用户明确要求“默认统一头部，异常时再回退到系统装饰”，则必须把“一体化默认路径”和“系统装饰 fallback 路径”都视为交付物的一部分；只实现 fallback 不算完成该请求。
- 如果目标平台无法同时提供统一头部视觉和窗管拥有的装饰行为，则默认保留系统标题栏，并把应用工具栏紧贴在标题栏下方，形成连续的头部布局。
- 当产品明确要求圆角、描边和投影由窗管接管时，不要把应用侧 `D.DWindow.windowRadius`、`borderWidth`、`borderColor`、`shadowRadius`、`shadowOffset`、`shadowColor` 的调校视为满足要求。
- 不要把无边框自绘窗口作为 Linux / UOS / Deepin 的通用默认方案。
- 对上述“双路径”请求，必须明确记录：验证所依据的 Qt/DTK 版本、Wayland/X11 会话、统一头部失效时切回系统装饰的触发条件，以及为何该 fallback 仍保留窗管装饰语义。
- 使用无边框或自绘标题栏前，至少确认以下条件：
  - 目标 Qt 版本支持 `startSystemMove()` / `startSystemResize()`
  - 目标会话下已经验证窗口拖动、缩放、最大化、还原、贴边和阴影表现
  - 窗口控制按钮具有键盘焦点和 `Accessible.name`
  - 该方案仍满足产品对窗管装饰效果和标准行为的要求；若不满足，必须回退到系统装饰
- Wayland 会比 X11 更严格地限制窗口移动、阴影和透明效果。默认把 Wayland 视为更保守的平台。

## 弹窗策略
- 通用默认值是 `Popup.Item`。它最稳定，也最容易跟随父窗口布局。
- 当菜单、下拉框或浮层需要由窗口管理器单独管理时，在 Qt 6.8 及以上可以考虑 `Popup.Window`。
- 使用 `Popup.Window` 时仍要提供 `Popup.Item` 回退路径，避免旧版本 Qt 或特定桌面环境失效。
- 不要把“窗口管理型弹窗”写成强制规则，应根据组件类型和目标平台决定。

## 模糊与透明
- 对承担桌面主导航的持久左侧栏，毛玻璃或等价 blur 是硬要求，不是可选增强。
- 对抽屉、浮层和其他非主导航区域，毛玻璃、实时模糊和半透明背景仍可视为渐进增强。
- 任何使用 `WindowBlur`、`MultiEffect` 或透明窗口的实现，都必须提供纯色背景回退。
- 对持久主导航左侧栏，如果目标环境无法提供已验证的 blur 路径，应把它当成平台阻塞点或取得显式用户豁免，而不是把纯色回退当成完成态。
- 在远程桌面、低性能设备、关闭特效或不支持 blur 的 compositor 下，可以自动退回到纯色面板，但这不改变“目标环境必须验证 blur”的交付要求。

## 主题与配色
- `Theme.mode` 默认使用 `system`，并跟随 `Qt.styleHints.colorScheme`。
- 优先将主题变量绑定到 DTK 或系统调色板；文档中的字面量色值仅作为 fallback。
- 当需要应用品牌色时，仅覆盖强调色相关变量，不要改写整套系统中性色。
- 导航、标签、步骤条和列表当前项的 hover / selected 颜色必须先进入主题层的语义变量，再由组件消费；不要在具体组件里动态混色来定义这些状态。

## 交互与无障碍
- 所有交互组件默认支持 `Tab` 聚焦、`Enter` / `Space` 激活、`Esc` 关闭相关浮层。
- 图标按钮至少提供 `Accessible.name`。
- 文本输入、下拉框、菜单和标签页需要可见焦点态。
- 自定义标题栏、菜单和侧边栏项必须能在无鼠标场景下完成主要操作。
- 对 16px 功能图标，要验证选中、悬停、禁用态是否与同级文字共享同一语义前景；如果图标来自网络下载的 SVG 资源，还要验证其风格是否与目标系统风格一致或接近一致，并保持 symbolic / alpha 可重着色语义，而不是已经被光栅化。

## 验证清单
- Qt / Qt Quick Controls 版本是否明确
- DTK 版本或目标发行版是否明确
- 目标会话是否为 Wayland 还是 X11
- 是否验证了系统移动、缩放、最大化、贴边、弹窗定位和关闭行为
- 是否确认顶层窗口的边框、阴影和圆角由窗管效果提供，或明确声明为何不能如此实现
- 是否确认实现没有继续通过应用侧 `D.DWindow` 装饰参数去补或改窗管应拥有的圆角、描边和投影
- 是否为 blur、透明和 popup window 提供了纯色或 `Popup.Item` 回退
- 对带持久主导航左侧栏的桌面应用，是否验证了控制中心式左右结构与左侧栏 blur 在目标环境中真实生效
- 应用主菜单是否由 DTK 标准菜单按钮触发，且弹出位置跟随 DTK 约定而不是手工坐标
- “关于”入口是否使用 DTK 标准 `AboutDialog`
- 如果用户要求默认统一头部，是否同时验证了统一头部默认路径与系统装饰 fallback 路径，而不是只保留 fallback
- 是否验证了 16px 功能图标在选中态与同级文字同色，且网络下载 SVG 路径没有把图标先光栅化成位图，同时图标风格与目标系统风格一致或接近一致
