# 桌面平台兼容性

## 目录
- DTK 优先
- 窗口策略
- 弹窗策略
- 模糊与透明
- 主题与配色
- 交互与无障碍
- 验证清单

## DTK 优先
- 优先使用 `org.deepin.dtk` 提供的窗口、按钮、菜单、对话框和表单控件。
- 只有当 DTK 缺少目标能力，或需要强定制化桌面壳层时，才退回自定义 QML 组件。
- 自定义组件必须保留系统主题同步、键盘导航、输入法兼容和无障碍信息。

## 窗口策略
- 默认保留系统标题栏和窗口装饰。这是 Linux 桌面上兼容性最高的方案。
- 仅在应用明确需要自定义桌面壳层时使用无边框窗口，例如浏览器式、三栏式或笔记类主窗口。
- 使用自绘标题栏前，至少确认以下条件：
  - 目标 Qt 版本支持 `startSystemMove()` / `startSystemResize()`
  - 目标会话下已经验证窗口拖动、最大化、还原和阴影表现
  - 窗口控制按钮具有键盘焦点和 `Accessible.name`
- Wayland 会比 X11 更严格地限制窗口移动、阴影和透明效果。默认把 Wayland 视为更保守的平台。

## 弹窗策略
- 通用默认值是 `Popup.Item`。它最稳定，也最容易跟随父窗口布局。
- 当菜单、下拉框或浮层需要由窗口管理器单独管理时，在 Qt 6.8 及以上可以考虑 `Popup.Window`。
- 使用 `Popup.Window` 时仍要提供 `Popup.Item` 回退路径，避免旧版本 Qt 或特定桌面环境失效。
- 不要把“窗口管理型弹窗”写成强制规则，应根据组件类型和目标平台决定。

## 模糊与透明
- 毛玻璃、实时模糊和半透明背景都是渐进增强，不是硬依赖。
- 任何使用 `WindowBlur`、`MultiEffect` 或透明窗口的实现，都必须提供纯色背景回退。
- 在远程桌面、低性能设备、关闭特效或不支持 blur 的 compositor 下，应自动退回到纯色面板。

## 主题与配色
- `Theme.mode` 默认使用 `system`，并跟随 `Qt.styleHints.colorScheme`。
- 优先将主题 token 绑定到 DTK 或系统调色板；文档中的字面量色值仅作为 fallback。
- 当需要应用品牌色时，仅覆盖强调色相关 token，不要改写整套系统中性色。

## 交互与无障碍
- 所有交互组件默认支持 `Tab` 聚焦、`Enter` / `Space` 激活、`Esc` 关闭相关浮层。
- 图标按钮至少提供 `Accessible.name`。
- 文本输入、下拉框、菜单和标签页需要可见焦点态。
- 自定义标题栏、菜单和侧边栏项必须能在无鼠标场景下完成主要操作。

## 验证清单
- Qt / Qt Quick Controls 版本是否明确
- DTK 版本或目标发行版是否明确
- 目标会话是否为 Wayland 还是 X11
- 是否验证了系统移动、最大化、弹窗定位和关闭行为
- 是否为 blur、透明和 popup window 提供了纯色或 `Popup.Item` 回退
