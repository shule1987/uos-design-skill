# 菜单组件

- 优先使用 DTK 菜单和菜单项。自定义菜单主要用于复杂 delegate、混排内容或平台控件不足的场景。

## MenuItem

```
┌─────────────────────────┐
│ 🔍 搜索         Ctrl+F  │
│ 📄 新建         Ctrl+N  │
│ 💾 保存         Ctrl+S  │
├─────────────────────────┤
│ ✓ 显示行号              │
│   自动换行              │
├─────────────────────────┤
│ ⚙ 设置                  │
└─────────────────────────┘
  图标  文本      快捷键
```

- 用途：承载单条菜单动作，可带图标、快捷键、勾选状态或危险语义。
- 结构：常见为“图标 / 文本 / 快捷键”三段式，也可以扩展为带说明或子菜单箭头的形式。
- 密度：菜单项高度保持统一，禁用态只降低可操作性和视觉权重，不改变整体节奏。
- 交互：支持 hover、键盘上下切换、`Enter` 激活和无障碍语义；勾选项与普通命令项要有清晰区分。

## PopupMenu
- 用途：用于承载一组菜单项、分组标题和分隔线。
- 视觉：背景优先使用 `Theme.popupBg`，边界和阴影只做轻量层级辅助。
- 行为：默认使用 `Popup.Item`，只有确有必要时才评估 `Popup.Window`；关闭后应返回焦点到触发控件。

## Application Main Menu

- 只要应用提供主菜单，主菜单就必须包含主题切换能力，不要把主题切换只留在设置页。
- 主题切换至少包含 `跟随系统`、`浅色`、`深色` 三项，名称可以本地化，但语义不可删减。
- `跟随系统` 不是一次性初始化动作，而是持续绑定系统主题状态；系统深浅主题变化后，应用应自动同步。
- 主题切换可以直接放在主菜单中，也可以放在主菜单的 `主题` 子菜单中；无论哪种形式，都要保持键盘可达、状态可见和当前项可判断。
- 当前主题项建议使用勾选、单选或等效 current-state 反馈，避免用户无法判断当前处于哪种模式。
