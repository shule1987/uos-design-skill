# 卡片组件

## Card

```
┌──────────────────────┐
│                      │
│    [缩略图区域]       │
│     (200x160)        │
│                      │
├──────────────────────┤
│  卡片标题             │
│                      │
│  这里是卡片的描述     │
│  文本内容...          │
│                      │
└──────────────────────┘
   (280x320, 带阴影)
```

- 用途：用于承载一组相对独立的信息块，如文档、素材、结果项和推荐内容。
- 结构：通常由缩略图、标题、描述和可选操作组成；但这些区域不是自由堆叠，而应先映射到明确的列、行、跨度和间距系统。
- 视觉：容器圆角使用 `Theme.radiusMd`，背景优先使用比窗口背景中性提亮约 `20%` 的 `Theme.cardBg`，只有在层级确实需要更重表面时才退到相近面板色；阴影保持轻量；卡片背景描边始终保持显式 `1px`，不要乘 UI scale 或 DPR 逻辑。
- 内边距底线：卡片背景边缘到真实内容之间的显式 inset 不得低于 `8px`；如果需要更紧凑，优先重排内部网格，不要继续压缩到贴边。
- 缩略图：卡片内若包含自动生成的展示结构布局缩略图，这类缩略图默认使用弱对比度、弱强调的处理，只承担方向提示，不得抢占标题、数值、按钮或其他真实交互元素的视觉重心。
- 缩略图边缘：所有结构缩略图都需要显式 `1px` 描边，直接保持 `1px`，不要再额外乘 UI scale 或设备像素比逻辑。
- 排版/结构预览：展示排版层级、文字节奏或布局骨架的缩略图，不要使用纯白或纯黑块去模拟内容，应使用弱对比度的预览墨色。
- 布局：卡片外部必须服从页面或卡片墙的响应式网格，先定义列数、跨度、最小宽度和 gutter，再摆放卡片；不要把不同尺寸卡片随意拼成边界不齐的松散墙面。
- 如果某类卡片组件本身依赖父级提供的行级等高协议，例如通过 `equalizedHeightForItem(...)` 从父级读取补齐高度，那么它必须挂在对应的响应式网格 primitive 下；不要把这类卡片直接塞进普通 `GridLayout` 后再假设高度约束仍然生效。
- 同一横向行里的卡片默认先顶对齐，再决定是否补齐高度；不要把本应组成一行的卡片在垂直方向错开、居中或做成不同起点。
- 同一视觉行里的相邻卡片，若高度差不足约 `40%`，无论处于双列卡片带还是多列响应式卡片墙，默认都应向该行较高卡片靠拢补齐高度，并保持顶部和底部外轮廓对齐；不要保留“几乎一样高但仍参差”的松散结果。明确跨多列的卡片默认可以保留独立高度，除非该组合另外要求整行共高。
- 双列横排卡片带里，如果较短那张卡片在补齐后会留下明显的大块垂直空白，不能把这视为“已经对齐”。当空白大致达到卡片主体高度的三分之一时，应视为组合失败，优先改网格而不是硬拉外壳。
- 处理这类双列不均衡时，优先顺序是：重分列跨度和断点、把较稀疏的一侧拆成上下两张卡片、或者给较稀疏卡片补一个真实的下半区块，例如摘要、状态、次级操作或列表；不要用纯 `Item` spacer、垂直居中或空白 filler 去撑满高度。
- 等高策略必须由真实内容区吸收，而不是由空白吸收。卡片内部若没有足够的内容层次去消化额外高度，就不应该继续维持这组等高外壳。
- 实现方式：这类双列卡片约束必须收敛到可复用的本地 primitive。恰好两张对等卡片的 band 使用类似 `EqualizedCardPairBand` 的组件；一侧更稠密、另一侧更稀疏或堆叠的双列组合使用类似 `BalancedTwoColumnCardBand` 的组件。
- 禁止在页面业务 QML 里直接写 `matchedCardHeight`、`Theme.equalizedCardPairHeight(...)` 或同类等高 glue。页面层只负责声明内容，等高和重排策略必须沉到复用组件里。
- 内部布局：卡片内部同样必须使用网格化编排。缩略图、标题、数值、说明、元信息和操作区要共享稳定对齐边界、基线和间距，不得通过随意塞 `Item`、魔法 margin 或无规律空白去拼视觉。
- 响应式：窗口变窄时，优先调整卡片列数、区域跨度和区块堆叠顺序，让内容按栅格重排；不要先缩小字号、压缩点击热区，或让关键元素脱离对齐系统。
- 交互：整张卡片可点击时，要和局部操作按钮区分交互热区，避免误触。
